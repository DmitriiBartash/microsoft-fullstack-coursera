Fundamentals of Logical Thinking in Programming

Practice the core habit every programmer relies on: turning a fuzzy requirement into a precise, step-by-step plan before writing real code. You will walk a small problem — "is this number even or odd?" — all the way from a plain-English definition through key processes, a flowchart, and finally pseudocode.

🎯 Objective

Learn to decompose a problem into ordered, unambiguous steps and express the solution as language-independent pseudocode, so the logic is clear and correct before it is translated into any specific programming language.

📦 What you will produce

A complete logical breakdown of one small program, captured in four artifacts that build on each other:

ArtifactWhat it captures
Problem definitionPlain-English statement of the inputs, the rule, and the outputs
Key processesThe ordered list of actions the program must perform
FlowchartA visual map of the control flow, including the decision branch
PseudocodeA structured, language-neutral version of the algorithm ready to code

Flow

Problem → Key processes → Flowchart → Pseudocode → (later) real code

🧭 Walkthrough

We will work the example end-to-end: a program that reads a number and reports whether it is even or odd.

1 Define the problem

State exactly what the program must do, in plain language, with no code.

Create a simple program that:

  • Accepts a number as input.
  • Checks if the number is even or odd.
  • Displays "Even number" if the number is divisible by 2, otherwise "Odd number".

A good definition names the input (a number), the rule (divisibility by 2), and the outputs (the two possible messages). Nail these down first — everything else follows from them.

2 Identify the key processes

List the actions the program performs, in order. This is the skeleton of the algorithm.

  • Start the program
  • Input a number
  • Check if the number is divisible by 2
  • If divisible, display "Even number"
  • Else, display "Odd number"
  • End the program

Notice the check introduces a decision — one input leads to two possible paths. Spotting where the logic branches is the whole point of this stage.

3 Draw the flowchart

Sketch the control flow visually. Each process becomes a shape, and the decision becomes a branch with two labelled exits (Yes / No).

        │     Start     │
        └───────┬───────┘
                │
        ┌───────▼───────┐
        │  Input Number │

                │ Yes                  │

4 Map the flowchart to pseudocode

Translate each flowchart element into a pseudocode construct. The mapping is deliberate — every symbol has a textual counterpart.

Flowchart elementPseudocode counterpart
StartStart of the process
Input NumberPrompt the user for input
DecisionUse an if condition to check divisibility
OutputPrint the appropriate message
EndEnd of the process

5 Write the pseudocode

Put it all together as structured, language-neutral pseudocode. Indentation shows which statements belong to the decision branches.

Start
    Prompt user to enter a number
    Read number
    If number modulo 2 equals 0 then
        Print "Even number"
    Else
        Print "Odd number"
    End If
End

This reads almost like English, yet maps one-to-one onto real code in any language — the hard thinking is already done.

✍️ Your turn

Apply the same five-stage process to a new problem of your own. A good practice target:

Read a number and report whether it is positive, negative, or zero.

Work through each stage in order:

  • Write the problem definition (input, rule, outputs).
  • List the key processes.
  • Draw the flowchart — note that this one needs two decisions, so it branches twice.
  • Map the flowchart to pseudocode elements.
  • Write the final pseudocode.

Keep the logic in pseudocode only — resist jumping to a programming language until the plan is complete and correct.

☑️ Definition of done

  • Problem is defined in plain English with input, rule, and outputs stated explicitly
  • Key processes are listed as an ordered sequence from start to end
  • Flowchart shows the control flow with the decision drawn as a labelled branch
  • Each flowchart element is mapped to its pseudocode counterpart
  • Final pseudocode uses If / Else / End If with correct indentation and is language-neutral

🔑 Key concepts

  • Decomposition first — break a requirement into the smallest ordered steps before writing any code; clear steps prevent confused code.
  • Pseudocode is language-independent — it expresses logic in structured plain language, so the same plan can be implemented in C#, Python, JavaScript, or any other language.
  • Decisions create branches — an if/else (here, number modulo 2 equals 0) splits one input into multiple paths; identifying these branches early is what flowcharts make visible.
  • Flowchart ↔ pseudocode are two views of one algorithm — every flowchart shape has a textual counterpart, so you can move between the visual and written forms freely.
  • Verify the logic before coding — getting the plan right up front is cheaper than debugging tangled code later.
