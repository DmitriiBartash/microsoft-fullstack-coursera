# Fundamentals of Logical Thinking in Programming

**Course 1 — Foundations of Coding Full-Stack** · Module 2 · Lesson 1 · `You Try It!`

> Practice the core habit every programmer relies on: turning a fuzzy requirement into
> a precise, step-by-step plan **before** writing real code. You will walk a small problem
> — *"is this number even or odd?"* — all the way from a plain-English definition through
> key processes, a flowchart, and finally **pseudocode**.

---

## 🎯 Objective

Learn to decompose a problem into ordered, unambiguous steps and express the solution as
**language-independent pseudocode**, so the logic is clear and correct *before* it is
translated into any specific programming language.

---

## 📦 What you will produce

A complete logical breakdown of one small program, captured in four artifacts that build
on each other:

| Artifact            | What it captures                                                        |
| ------------------- | ----------------------------------------------------------------------- |
| Problem definition  | Plain-English statement of the inputs, the rule, and the outputs        |
| Key processes       | The ordered list of actions the program must perform                    |
| Flowchart           | A visual map of the control flow, including the decision branch         |
| Pseudocode          | A structured, language-neutral version of the algorithm ready to code   |

**Flow:** `Problem → Key processes → Flowchart → Pseudocode → (later) real code`

---

## 🧭 Walkthrough

We will work the example end-to-end: a program that reads a number and reports whether it
is **even** or **odd**.

### Step 1 — Define the problem

State exactly what the program must do, in plain language, with no code:

Create a simple program that:

- Accepts a number as input.
- Checks if the number is even or odd.
- Displays `"Even number"` if the number is divisible by 2, otherwise `"Odd number"`.

A good definition names the **input** (a number), the **rule** (divisibility by 2), and
the **outputs** (the two possible messages). Nail these down first — everything else
follows from them.

### Step 2 — Identify the key processes

List the actions the program performs, in order. This is the skeleton of the algorithm:

1. Start the program
2. Input a number
3. Check if the number is divisible by 2
4. If divisible, display `"Even number"`
5. Else, display `"Odd number"`
6. End the program

Notice step 3 introduces a **decision** — one input leads to two possible paths. Spotting
where the logic branches is the whole point of this stage.

### Step 3 — Draw the flowchart

Sketch the control flow visually. Each process becomes a shape, and the decision becomes a
branch with two labelled exits (`Yes` / `No`):

```text
        ┌───────────────┐
        │     Start     │
        └───────┬───────┘
                │
        ┌───────▼───────┐
        │  Input Number │
        └───────┬───────┘
                │
        ╱───────▼───────╲
       ╱  number % 2     ╲   No
       ╲   == 0 ?        ╱ ────────────┐
        ╲───────┬───────╱              │
                │ Yes                  │
        ┌───────▼───────┐     ┌────────▼────────┐
        │ Print         │     │ Print           │
        │ "Even number" │     │ "Odd number"    │
        └───────┬───────┘     └────────┬────────┘
                │                      │
                └──────────┬───────────┘
                           │
                   ┌───────▼───────┐
                   │      End      │
                   └───────────────┘
```

> *(The original includes a flowchart diagram here; the version above is a text
> representation of the same control flow.)*

### Step 4 — Map the flowchart to pseudocode

Translate each flowchart element into a pseudocode construct. The mapping is deliberate —
every symbol has a textual counterpart:

| Flowchart element | Pseudocode counterpart                          |
| ----------------- | ----------------------------------------------- |
| Start             | Start of the process                            |
| Input Number      | Prompt the user for input                       |
| Decision          | Use an `if` condition to check divisibility     |
| Output            | Print the appropriate message                   |
| End               | End of the process                              |

### Step 5 — Write the pseudocode

Put it all together as structured, language-neutral pseudocode. Indentation shows which
statements belong to the decision branches:

```text
Start
    Prompt user to enter a number
    Read number
    If number modulo 2 equals 0 then
        Print "Even number"
    Else
        Print "Odd number"
    End If
End
```

This reads almost like English, yet maps one-to-one onto real code in any language — the
hard thinking is already done.

## ✍️ Your turn

Apply the same five-stage process to a new problem of your own. A good practice target:

> Read a number and report whether it is **positive**, **negative**, or **zero**.

Work through each stage in order:

- Write the **problem definition** (input, rule, outputs).
- List the **key processes**.
- Draw the **flowchart** — note that this one needs *two* decisions, so it branches twice.
- Map the flowchart to **pseudocode** elements.
- Write the final **pseudocode**.

Keep the logic in pseudocode only — resist jumping to a programming language until the
plan is complete and correct.

- [ ] Each flowchart element is mapped to its pseudocode counterpart

## 🔑 Key concepts

- **Decomposition first** — break a requirement into the smallest ordered steps before
  writing any code; clear steps prevent confused code.
- **Pseudocode is language-independent** — it expresses logic in structured plain language,
  so the same plan can be implemented in C#, Python, JavaScript, or any other language.
- **Decisions create branches** — an `if`/`else` (here, `number modulo 2 equals 0`) splits
  one input into multiple paths; identifying these branches early is what flowcharts make
  visible.
- **Flowchart ↔ pseudocode are two views of one algorithm** — every flowchart shape has a
  textual counterpart, so you can move between the visual and written forms freely.
- **Verify the logic before coding** — getting the plan right up front is cheaper than
  debugging tangled code later.
