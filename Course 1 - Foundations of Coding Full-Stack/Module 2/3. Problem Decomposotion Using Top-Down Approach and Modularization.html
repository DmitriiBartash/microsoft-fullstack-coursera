Problem Decomposition Using a Top-Down Approach and Modularization

Take two real-world systems — a fitness tracking app and an online learning platform — and break each one down with the top-down approach: start from a single goal, split it into major features, decompose those into concrete tasks, then map the result onto independent modules. The point is the method, not the apps themselves.

🎯 Objective

Practice decomposing a large problem into manageable pieces using top-down design, and then modularizing the result so each logical area becomes a standalone unit that can be built, tested, and reused independently.

🧩 What you will produce

For each of the two problems below, a four-part decomposition:

ArtifactWhat it captures
Main goalOne sentence stating what the system is for
Major featuresThe top-level functional areas (the first top-down split)
Decomposed tasksEach feature broken into concrete, manageable sub-tasks
Module mapA table assigning each area to an independent module + its responsibility

Method

Goal → Major features → Sub-tasks per feature → Modules

🚶 Walkthrough — Problem 1: Fitness Tracking App

Statement: Decompose the development of a fitness tracking app that can track users' workouts, monitor their diet, and provide health insights.

1 Define the main goal

Build a fitness tracking app that helps users log and monitor workouts, track diet and nutrition, and receive personalized health insights.

2 Identify the major features (top-down, level 1)

Break the app into its main functional areas:

  • Workout Tracking
  • Diet Monitoring
  • Health Insights
  • User Management

3 Decompose each feature (top-down, level 2)

1. Workout Tracking

  • Add a new workout (type, duration, calories)
  • View workout history
  • Plan future workouts (calendar, goals)
  • Display workout progress (charts, summaries)

2. Diet Monitoring

  • Log daily meals and snacks
  • Search food database (name, brand, macros)
  • Track calories and macronutrients (carbs, proteins, fats)
  • Monitor water intake

3. Health Insights

  • Generate weekly/monthly reports
  • Suggest personalized recommendations (e.g., increase protein intake, reduce sugar)
  • Set and track health goals (weight loss, muscle gain, maintenance)
  • Integrate with fitness wearables (e.g., Fitbit, Apple Watch)

4. User Management

  • User registration and login
  • Store personal data (age, height, weight, activity level)
  • Set personal preferences and goals
  • Manage app settings and notifications

4 Modularize

Each core feature becomes a separate module, enabling independent development and testing:

ModuleResponsibility
WorkoutModuleAdd, view, plan, and analyze workouts
NutritionModuleLog meals, calculate calories/macros, manage food database
HealthInsightsModuleGenerate reports, offer health recommendations
UserModuleHandle login, profile, preferences
DataStorageModuleStore and retrieve workout/diet/user data
AnalyticsModuleProduce graphs, summaries, and track progress
IntegrationModuleOptional: sync with external APIs and wearables

In short: define the broad goal → use top-down design to find functional areas → decompose each into sub-tasks → modularize each part into an independent unit for clean, maintainable, scalable code.

🚶 Walkthrough — Problem 2: Online Learning Platform

Statement: Decompose the creation of an online learning platform that supports course creation, user enrollment, content delivery, and progress tracking.

1 Define the main goal

Build an online learning platform where instructors create and manage courses, students enroll and access content, and the system delivers content, tracks progress, and supports learning.

2 Identify the major features (top-down, level 1)

Divide the platform into key functional components:

  • Course Management
  • User Enrollment & Authentication
  • Content Delivery
  • Progress Tracking
  • Communication & Interaction (optional)
  • Admin Panel (optional)

3 Decompose each feature (top-down, level 2)

1. Course Management (for instructors)

  • Create / edit / delete courses
  • Add modules, lessons, and quizzes
  • Upload files (PDFs, videos, slides)
  • Set course requirements and schedules

2. User Enrollment & Authentication

  • User registration (students & instructors)
  • Login / logout and role-based access
  • Enroll or unenroll from courses
  • View enrolled-courses dashboard

3. Content Delivery

  • Stream video lessons
  • Display text materials and resources
  • Provide download links for files
  • Structure content by modules or sections

4. Progress Tracking

  • Track completed lessons/modules
  • Display course progress bar
  • Show quiz results or scores
  • Provide certificates (optional)

5. Communication & Interaction (optional)

  • Discussion forums per course
  • Live Q&A or chat functionality
  • Instructor feedback on assignments

6. Admin Panel (optional)

  • Manage users and permissions
  • Monitor course statistics and activity
  • Moderate content and discussions

4 Modularize

Each major feature is built as a standalone module, letting developers work independently and reuse components:

ModuleResponsibility
CourseModuleHandles course creation, structure, content upload
UserModuleRegistration, authentication, and user roles
EnrollmentModuleCourse enrollment management
ContentDeliveryModuleStreaming, displaying, and organizing course materials
ProgressModuleTracks student progress and quiz performance
CommunicationModuleForums, messaging, and feedback systems (optional)
AdminModulePlatform-wide settings, user/course moderation (optional)
StorageModuleHandles file uploads and downloads

In short: define the main goal → top-down split into broad feature areas → decompose features into detailed tasks (e.g., lessons under courses, registration under user management) → modularize each logical area for cleaner architecture, scalability, and parallel development.

✍️ Your turn

Apply the same four-step method to a system of your own. Pick a domain (e.g., a recipe manager, an expense tracker, a ride-sharing app) and produce:

  • A one-sentence main goal.
  • The major features from a top-down split.
  • Each feature decomposed into concrete sub-tasks.
  • A module map table (Module → Responsibility), including any cross-cutting modules (storage, analytics, integrations) that several features share.

🌟 What good looks like

  • Every system starts from a single, clearly stated goal.
  • The first split is top-down: a few coarse feature areas, not a flat list of tasks.
  • Each feature is decomposed into small, concrete, actionable sub-tasks.
  • A module map assigns each area to an independent module with a one-line responsibility.
  • Cross-cutting concerns (storage, analytics, integrations) are pulled out into their own shared modules instead of being duplicated per feature.
  • Modules have single, non-overlapping responsibilities so they can be developed, tested, and reused independently.

🔑 Key concepts

  • Top-down decomposition — start from the whole, then progressively split it into smaller parts (goal → features → tasks). You stop refining when each leaf is small enough to build directly.
  • Modularization — group related tasks into a module with one clear responsibility, so the system becomes a set of loosely coupled, independently testable units.
  • Single responsibility — a well-named module (WorkoutModule, EnrollmentModule) should have exactly one reason to change; overlapping responsibilities are a smell.
  • Cross-cutting modules — concerns shared across features (storage, analytics, third-party integrations) become their own modules so logic isn't duplicated.
  • Parallel, scalable development — clean module boundaries let different developers work in parallel and let the system grow without the codebase becoming tangled.
