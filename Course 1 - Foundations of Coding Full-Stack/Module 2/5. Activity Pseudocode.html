Activity: Pseudocode

Practise turning a plain-English problem into pseudocode — language-agnostic, step-by-step logic — before writing any real code. You follow a fixed three-part method (define → identify processes → write pseudocode) and apply it to two short problems built around the Input → Processing → Output model.

🎯 Objective

Learn to translate a problem statement into clear pseudocode by defining the problem, identifying its key processes, and expressing the logic as simple, ordered steps — independent of any programming language.

📦 What you will produce

For each problem, a short worked solution in three parts:

PartWhat it captures
1. Define the problemA one-sentence restatement of what the program must do
2. Identify processesThe Input, Processing, and Output the problem needs
3. PseudocodeOrdered, plain steps between Start and End

Model

Input → Processing → Output

🧭 Walkthrough

Two problems are worked end-to-end below. Notice how each one follows the same define → identify → pseudocode rhythm.

1 Problem 1 — Calculate Total Marks

Write pseudocode for a program that calculates the total marks for a student across three subjects — Math, Science, and English. The program takes the marks for each subject as input, calculates the total, and prints it.

1. Define the problem

You need a program that takes the marks for three subjects (Math, Science, English) as input, calculates the total marks, and prints the result.

2. Identify key processes

  • Input: Get marks for Math, Science, and English.
  • Processing: Add the three marks to compute the total.
  • Output: Display the total marks.

3. Pseudocode

Start
    Get Math marks
    Get Science marks
    Get English marks
    Set total to Math marks + Science marks + English marks
    Print total
End

2 Problem 2 — Greet the User by Name

Write pseudocode for a program that takes a user's name as input and greets them with a message that says "Hello, [name]!".

1. Define the problem

Write a program that asks for the user's name and prints a greeting message in the format: "Hello, [name]!".

2. Identify key processes

  • Input: Get the user's name.
  • Processing: Construct the greeting message.
  • Output: Display the greeting message.

3. Pseudocode

Start
    Get user's name
    Set greeting to "Hello, " + user's name + "!"
    Print greeting
End

✍️ Your turn

Pick a small problem of your own and run it through the same three-part method:

  • Define the problem in one clear sentence.
  • Identify the processes — what is the Input, the Processing, and the Output?
  • Write the pseudocode as ordered steps wrapped in Start … End.

A good practice problem: calculate the average of the three subject marks from Problem 1 (reuse the inputs, then divide the total by 3 before printing).

👍 What good looks like

A strong solution restates each problem in one sentence, lists Input / Processing / Output separately, and expresses language-agnostic steps with plain verbs (Get, Set, Print) wrapped in Start … End — so any reader could translate it into the language of their choice.

☑️ Definition of done

  • Problem 1 has all three parts: definition, identified processes, and pseudocode
  • Problem 2 has all three parts: definition, identified processes, and pseudocode
  • Both pseudocode blocks correctly use Get, Set, and Print between Start and End
  • Total in Problem 1 is the sum of the three subject marks
  • Greeting in Problem 2 matches the exact format "Hello, [name]!"
  • You have written and checked one pseudocode solution of your own

🔑 Key concepts

  • Pseudocode is language-agnostic — it describes logic in plain steps, so the same solution can be implemented in any programming language.
  • Define before you design — restating the problem in one sentence clarifies exactly what the program must accomplish before you write a single step.
  • Input → Processing → Output — almost every small program decomposes into getting data, transforming it, and showing a result; spotting these three stages structures the logic.
  • Plain, ordered actions — verbs like Get, Set, and Print, one per line between Start and End, keep the flow unambiguous.
  • A repeatable method scales — the same define → identify → write approach works for a two-line greeting or a far larger program.
