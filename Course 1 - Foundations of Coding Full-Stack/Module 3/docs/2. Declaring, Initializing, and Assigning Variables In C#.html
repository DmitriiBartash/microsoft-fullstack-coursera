Declaring, Initializing, and Assigning Variables In C#

Work through two short problems that practice the core habit of every C# program: picking the right data type, giving a variable a meaningful name, and initializing it with the = operator. You'll declare a double for a product price and a bool for an admin flag, then check your reasoning against the explanation.

🎯 Objective

Be able to declare, initialize, and assign variables in C# by choosing the appropriate data type for the value, naming the variable so it reflects what it stores, and using the assignment operator (=) to set its initial value.

📦 What you'll produce

Two correct, idiomatic C# variable declarations, each justified by a short rationale.

#GoalTypeDeclaration
1Store a product's price (49.99)doubledouble productPrice = 49.99;
2Flag whether a user is an admin (true)boolbool isAdmin = true;

Each declaration follows the same shape:

type name = value;
  • type — the data type that fits the value (double, bool, int, string, …).
  • name — a descriptive identifier (camelCase for locals in C#).
  • value — the literal assigned with the = operator, ending in a semicolon.

✅ Prerequisites

  • .NET SDK installed — check with dotnet --version
  • Visual Studio Code (or any C# editor)
  • Familiarity with C# literal syntax for numbers and Booleans

🧭 Walkthrough

1 Declaring and initializing a double

Declare a variable to store the price of a product and initialize it with a value of 49.99.

Instructions

  • Identify the appropriate data type for the variable.
  • Declare the variable with an appropriate name.
  • Initialize the variable with the value 49.99.

Template

____ productPrice = ____;

Solution

double productPrice = 49.99;

Why this works

  • double is the appropriate data type for values with decimals.
  • productPrice is a meaningful variable name that reflects what the value represents.
  • 49.99 is assigned using the = operator.

2 Declaring and initializing a bool

Declare a variable to store a flag indicating whether a user is an admin, and initialize it with the value true.

Instructions

  • Identify the appropriate data type for the variable.
  • Declare the variable with an appropriate name.
  • Initialize the variable with the value true.

Template

____ isAdmin = ____;

Solution

bool isAdmin = true;

Why this works

  • bool is the appropriate data type for true/false (Boolean) values.
  • isAdmin is a descriptive variable name indicating what the Boolean represents.
  • true is assigned using the = operator to indicate that the user is an admin.

🔁 Your turn

Put both declarations together and print them, so you can see the values at runtime:

double productPrice = 49.99;
bool isAdmin = true;

Console.WriteLine($"Product price: {productPrice}");
Console.WriteLine($"Is admin: {isAdmin}");

Run it from a console project:

dotnet run

Then try a few variations to cement the pattern:

  • Declare an int for a quantity in stock, e.g. int stockCount = 120;.
  • Declare a string for a product name, e.g. string productName = "Wireless Mouse";.
  • Ask yourself: would int work for 49.99? (No — it would drop the decimal part.)

🌟 What good looks like

  • Each variable uses the smallest, most fitting data type for its value (double for 49.99, bool for true).
  • Variable names are descriptive camelCase identifiers (productPrice, isAdmin) — not x or temp.
  • Every declaration initializes the variable in the same statement with = and ends in a semicolon.
  • You can explain why each type was chosen, not just that it compiles.

🔑 Key concepts

  • Type fits the value — pick the data type from the kind of data: decimals → double, true/false → bool, whole numbers → int, text → string.
  • Declare = type + name — a declaration introduces a variable by stating its type and identifier; C# is statically typed, so the type is fixed once declared.
  • Initialize with = — the assignment operator binds a value to the variable; declaring and initializing in one line is the common, readable form.
  • Names communicate intent — productPrice and isAdmin tell the next reader what the value means; good names are the cheapest documentation you can write.
