Planning Security for API Endpoints — Solutions

Worked answers for the two student tasks. Each route is planned on three axes: JWT validation (authentication — who you are), roles (coarse authorization — what type of thing you may do), and claims (fine authorization — which specific record you may touch).

Reading the claim column. "Must include the user ID" means the caller's own identity from the token is compared against the resource's owner — the token does not carry a list of every record the caller may reach.

📱 Task 1 — Social Media Platform

1 Scenario

A social media platform exposes:

  • GET /users/{id} — returns profile information for a specific user.
  • POST /posts — creates a new post.
  • DELETE /posts/{id} — deletes a specific post.

Step 1 — Which routes require JWT validation?

  • GET /users/{id} — JWT required. A profile holds personal information and is subject to privacy settings and block lists, so the platform must know who is asking before deciding what to return; requiring a token also blocks anonymous scraping. (If the platform intentionally publishes fully public profiles, a trimmed version could be AllowAnonymous; the secure default is to authenticate, then filter fields by the viewer's relationship to the target.)
  • POST /posts — JWT required. Creating content is a write that must be tied to a real, authenticated author; anonymous posting invites spam and makes authors un-attributable.
  • DELETE /posts/{id} — JWT required. Deletion is a destructive write on a specific record; only an authenticated user may attempt it, and only the owner or a moderator may succeed.
All three routes require JWT validation. Unlike an e-commerce product catalog, none of these operations is safely anonymous.

Step 2 — Specify roles and claims

  • GET /users/{id} — Roles: user (any authenticated user). Viewing profiles is the core of a social network, so this route is intentionally not owner-restricted — you view other people's profiles. Claims: the caller's user ID applies relationship rules (privacy, blocks, friends-only fields), not an owner restriction. This is the key contrast with GET /orders/{id}, which is owner-only.
  • POST /posts — Roles: user. Claims: must include the caller's user ID, used to set the new post's author. The client must never send an arbitrary authorId — it is taken from the token so a post is created only on the caller's behalf.
  • DELETE /posts/{id} — Roles: user (as the post's owner) or moderator / admin. Claims: the caller's user ID must match the post's authorId or the caller holds a moderation role — the "owner or elevated role" pattern.

Why this design

  • JWT decisions. Every route either exposes personal data or performs a write, so none is a candidate for anonymous access — authenticate-first, the opposite of a public product catalog.
  • Roles & claims. GET /users/{id} stays open to any authenticated user because a social graph exists to view others; owner-scoping it would break the product. Writes tighten access: the author comes from the token (never the body), and delete adds "owner or moderator" so moderation is possible without letting users delete each other's posts.
  • Data protection. The author claim prevents impersonation, the ownership check on delete prevents tampering, and JWT validation keeps the whole surface off-limits to anonymous abuse — defense in depth.

🎓 Task 2 — Learning Management System (LMS)

2 Scenario

An LMS exposes:

  • GET /courses — lists all available courses.
  • POST /assignments — creates a new assignment.
  • GET /grades/{id} — returns grades for a specific student.

Step 1 — Which routes require JWT validation?

  • GET /courses — JWT not required (public). A catalog of available courses is marketing surface meant to be browsable by prospective students; like GET /products, it carries no sensitive data. (If courses are private to an enrolled institution, this becomes [Authorize] and lists only the caller's available courses — but the default catalog is public.)
  • POST /assignments — JWT required. Authoring an assignment is a privileged write; restricted to authenticated staff who own the relevant course.
  • GET /grades/{id} — JWT required. Grades are highly sensitive academic records (protected under privacy regulations such as FERPA); access must be authenticated and then tightly scoped.
Two of three routes require JWT validation; the public catalog mirrors the e-commerce products example.

Step 2 — Specify roles and claims

  • GET /courses — Roles: none (public). Claims: none.
  • POST /assignments — Roles: instructor, admin (students must never create assignments). Claims: the instructor must own/teach the target course — the course's owner is checked against the caller's ID; an admin may create for any course. A resource-ownership check on the course the assignment belongs to.
  • GET /grades/{id} — Roles: student (owner of the record), instructor, admin. Claims:
    • student — caller's user ID must match {id}; sees only their own grades.
    • instructor — limited to students enrolled in courses they teach, not the whole student body.
    • admin — full access for administrative purposes.

Step 3 — Security plan summary table

RouteJWT validationRolesClaims
GET /coursesNot requiredNone (public)None.
POST /assignmentsRequiredinstructor, adminInstructor must own the target course; admin unrestricted.
GET /grades/{id}Requiredstudent, instructor, adminStudent: ID must match {id}. Instructor: own students only. Admin: full.

Why this design

  • JWT decisions. The catalog is public to encourage discovery and enrollment, like a product list. Everything that authors content or exposes grades is gated.
  • Roles & claims. POST /assignments shows role + resource ownership: the role gets you to the route, but a course-ownership check stops one instructor adding assignments to another's course. GET /grades/{id} shows one resource, three scopes: students (own grades), instructors (own students), admins (all) — each role narrowed by a different claim/relationship rule.
  • Data protection. Grades — the most sensitive asset — are reachable only by the owning student, the responsible instructor, or an admin (least privilege, FERPA). Assignment authoring is walled off from students and scoped to course ownership, while the public catalog deliberately exposes nothing sensitive.

🔧 Enforcement at a glance (ASP.NET Core)

The role gate is declarative ([Authorize(Roles = …)]); the ownership / claim check (axis 3) is imperative, because the framework can't know who owns record {id} until it loads it.

// GET /courses  — public
[AllowAnonymous]
[HttpGet("/courses")]
public IActionResult ListCourses() => Ok(_catalog.All());

// POST /assignments — instructors or admins; ownership checked imperatively
[Authorize(Roles = "instructor,admin")]
[HttpPost("/assignments")]
public IActionResult Create(CreateAssignment dto)
{
    var callerId = User.FindFirst(ClaimTypes.NameIdentifier)!.Value;
    if (!User.IsInRole("admin") && !_courses.IsOwnedBy(dto.CourseId, callerId))
        return Forbid();                       // axis 3: resource ownership
    return Ok(_assignments.Create(dto));
}

// GET /grades/{id} — self, instructor-of-record, or admin
[Authorize(Roles = "student,instructor,admin")]
[HttpGet("/grades/{id}")]
public IActionResult Grades(string id)
{
    var callerId = User.FindFirst(ClaimTypes.NameIdentifier)!.Value;
    if (User.IsInRole("student") && callerId != id) return Forbid();   // own grades only
    if (User.IsInRole("instructor") && !_enroll.Teaches(callerId, id)) return Forbid();
    return Ok(_grades.For(id));
}

For reuse, the ownership logic can move into a resource-based IAuthorizationHandler behind a named policy.
